# Event Setup Tool
**Category:** Product Design — Employer: Meta

---

## Case Study

Setting up a Facebook Pixel correctly meant finding every button, link, page view, and checkout on a site and manually adding tracking code to each one. Miss a spot — and there were many spots to miss — and the data was incomplete. For marketers without engineering support, it was a copy-paste-heavy process that was easy to get wrong and hard to verify.

The solution was to remove the code entirely from the process. The team modified the Pixel's base code so a tool could load as a layer directly on top of any website, identify all trackable elements automatically, and let the user assign events by simply clicking on them. Parameters worked the same way: select the on-screen text representing a price, a size, or a color, and the tool read the underlying code — the specific span or class ID — and grabbed that value directly. No manual updates to the site needed. It read what was already there.

I joined the project near the start, after another designer had established the basic foundation, and took ownership of the parameter setup experience — extending the tool into meaningfully more complex territory while keeping the overall flow simple. I ran multiple rounds of usability testing with a UX researcher using the RITE method, which let us move quickly and refine the experience in real time. I also cleaned up the visual design and made the overlay movable on the page.

The tool became the primary method for Pixel event setup for several years — the preferred approach both for users and internally at Facebook — until the industry's shift to server-side tracking.

## Raw Notes (verbatim)

The event setup tool is the first main project I worked on during my time at Facebook. The problem was that setting up a pixel was really hard because you needed to go find every button or link or page view or checkout and set up code on each and every single one of them so that you could set your pixel up correctly. That meant that there were so many places that you might miss and so much code you had to do copy and paste and it became really hard for marketers to do that. The solution was to provide a no-code solution. The team modified the pixel base code so that you could load a tool on top of your website that identified all of the buttons and links and necessary items and allowed you to click on them and assign them an event. This also made setting up parameters easy because all you had to do was select the text on the page that was the price or the size or the color and then we would look at the code and grab the stuff from that specific span from that specific class ID. You didn't have to re-update your code anymore. We just read your site for you. I came onto the team near the very beginning after a designer had spent a little bit of time getting the basics down. I then took it into parameters and added more additional complexity while keeping the overall flow very simple. I also ran multiple tests with one of our UX researchers, including a rise method, write method, or whatever, the rapid iterative testing, so that we could really refine it and take it further and clean it up. I cleaned up the visuals, made it movable, all of that stuff. It became the main way that people set up events on their pixel for several years ahead of the move to server-side. It became the preferred way of setting up events both by users and by Facebook.
